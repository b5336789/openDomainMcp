Technical Documentation

openDomainMcp 技術文件

本文件提供 openDomainMcp 的工程參考，內容涵蓋整體架構與資料模型、攝取管線、程式碼的語法樹切塊， 以及嵌入、向量儲存，與 REST、MCP、命令列三種對外介面的規格。

Python ≥ 3.11Chroma · cosinetree-sitter fastembed · 384dhybrid · BM25+RRF RAG · askmulti-collectionFastAPI · SSE · MCP

01系統概觀

openDomainMcp 接收各種型態的文件與多種程式語言的原始碼，從中整理出領域知識， 並存入向量資料庫供語意檢索。整體設計圍繞以下幾個原則：

  • 單一事實來源。命令列、MCP 伺服器與 Web API 共用同一份執行情境 （Context，由設定、向量庫與管線組成），三者行為必然一致。
  • 依語意切塊。程式碼沿語法樹的邊界切分（作法借鏡 claude-context），文件則以遞迴方式切分。
  • 以知識增強檢索。每段內容先由 Claude 產生結構化摘要，再折入嵌入文字，使檢索貼近語意而非字面。
  • 明確回報，絕不靜默丟棄。略過與錯誤一律記錄並回傳，資料不會無聲消失。
  • 可替換、可注入。嵌入後端、萃取器與切塊器皆可抽換，測試以替身元件全程離線執行。

02架構與資料流

攝取流程是一條線性的六階段管線。三個前端介面並不各自實作邏輯，而是一律透過 build_context() 取得共用元件：

┌─────────── 介面層 (interchangeable front-ends) ───────────┐ │ cli.py server.py (MCP) api/app.py (FastAPI) │ └───────────────┬──────────────┬───────────────┬────────────┘ └──────────────┴───────────────┘ context.build_context() ┌──────────────┴───────────────┐ │ Context(settings, store, pipeline) └──────────────┬───────────────┘ ▼ load ──▶ split ──▶ extract ──▶ embed ──▶ store ──▶ search loader code/text knowledge embedder Chroma Chroma splitter (Claude) (vector) (upsert) (query)

所有路徑都建立在 opendomainmcp/context.py 的工廠之上，因此系統裡只會有一份管線實作。

03資料模型

定義於 models.py，皆為不含業務邏輯的 dataclass，於各層之間傳遞。

Chunk

嵌入與儲存的基本單位。kind 為 "code" 或 "text"；程式碼片段會額外攜帶語法樹的中繼資料。

欄位型別說明
textstr原始片段內容。
sourcestr來源檔案路徑。
kindstrcode 或 text。
languagestr?程式語言名稱（文件為 None）。
node_typestr?語法樹節點型別，或 block / lines。
symbolstr?定義的符號名稱（函式或類別名）。
start_line / end_lineint?從 1 起算的起訖行號。
knowledgeKnowledgeUnit?萃取所得的領域知識。

衍生行為：

  • content_hash / id：取 sha256("{source}:{start}-{end}\n{text}")，作為冪等 upsert 的穩定識別碼。
  • embedding_text()：若已有知識，回傳「原文＋摘要＋概念」的增強文字；否則回傳原文。
  • metadata()：扁平化、且去除 None 的中繼資料字典，方便寫入 Chroma。

KnowledgeUnit / SearchResult

class KnowledgeUnit:  summary: str; concepts: list[str]; relations: list[str]
class SearchResult:  id: str; text: str; score: float; metadata: dict

04攝取管線

Pipeline.ingest_path(path, progress) 接受單一檔案或目錄。傳入目錄時會遞迴走訪， 並略過版本控制與相依目錄（.git、node_modules、.venv、 __pycache__、dist、build 等）以及隱藏檔。

逐檔處理與進度事件

處理過程透過選用的 progress 回呼逐步發出事件，Web 端即依此以 SSE 串流即時狀態：

stage意義
load開始載入檔案。
split切塊中（程式碼走語法樹，文件走遞迴）。
extract逐段萃取知識，detail 帶有片段數。
embed / store向量化後 upsert 入庫。
prune移除該來源的孤兒片段，或目錄同步時移除已刪除檔案的片段。
skip不支援、二進位或無內容（記入 report.skipped）。
error非預期錯誤（記入 report.errors，流程繼續）。
done / report整體完成；report 帶有彙總計數。
明確回報：UnsupportedFileError 歸入 skipped， 其他例外歸入 errors。知識萃取屬盡力而為，單一片段失敗只記錄、不中斷整體攝取。 最終回傳 IngestReport(files_indexed, chunks_indexed, chunks_pruned, skipped[], errors[])。 變更感知同步與並行萃取詳見第 16 節。

05語法樹程式碼切塊

split_code(source, language, path, max_chars) 借鏡 claude-context 的核心想法： 先以 tree-sitter 解析，再沿語意邊界（函式、類別、方法）切塊，而非任意的字元視窗。 各語言的 grammar 內建於對應的 tree-sitter-* wheel，離線即可使用，解析器並會快取重用。

切塊邏輯

  1. 同層走訪（_chunk_siblings）：依序檢視節點。連續的非定義節點（如 import、敘述） 會先行合併，直到累積跨度超過 max_chars 才切出一塊。
  2. 定義節點：遇到 _DEFINITION_TYPES（如 function_definition、 class_declaration、impl_item 等）時，先沖出當前群組，再單獨成塊， 並標註 symbol（取 name 欄位，或第一個 identifier）。
  3. 過大的定義（_split_definition）：若單一定義超過 max_chars， 便遞迴進入其主體（block、class_body、declaration_list 等）切分成員； 若找不到可辨識的主體，則整塊保留，以免遺失符號名稱。
  4. 退回機制：當語言缺少對應 grammar、解析失敗，或結果為空（空檔或僅含註解）時， 改用 _line_fallback 以行區間切塊，節點型別記為 lines，同時寫下警告。

支援語言對應

Python、JavaScript、TypeScript、TSX、Java、Go、Rust、C、C++、C#、Ruby、Bash。 語言名稱對應至 tree_sitter_* 模組及其 language*() 工廠函式 （TypeScript 較特別，分為 language_typescript 與 language_tsx）。

# 每一塊都帶有可追溯的定位資訊
Chunk(kind="code", language="python", node_type="function_definition",
      symbol="build_context", start_line=25, end_line=36)

06文件載入與文字切塊

載入與型別偵測（loader.py）

  • 程式碼副檔名對應為 LoadedDoc(kind="code", language=...)。
  • .pdf 以 pypdf 抽取、.docx 以 python-docx 抽取、.html 以 HTMLParser 抽取（並略過 script 與 style）。
  • 已知的文字副檔名（txt、md、json、yaml、csv、xml、css 等）直接讀取。
  • 未知副檔名則先嘗試以 UTF-8 解碼：成功便視為文字，失敗則丟出 UnsupportedFileError（通常代表二進位檔）。

遞迴文字切塊（text_splitter.py）

RecursiveTextSplitter(chunk_size, chunk_overlap) 依分隔符的優先序 ["\n\n","\n",". "," ",""] 由粗到細處理：先用文本中存在的最粗分隔符切開， 對仍然過長的片段往下遞迴，最後再把小片段合併成約 chunk_size 大小的塊， 並保留 chunk_overlap 字元的前文重疊。若 chunk_overlap ≥ chunk_size 會直接丟出 ValueError。

07知識萃取

萃取器的介面為 extract(text, kind, language) -> KnowledgeUnit， 由 get_extractor(settings) 依設定決定實作：

  • ClaudeExtractor：呼叫 Anthropic 的 messages.create。模型由 ODM_EXTRACTION_MODEL 指定（預設 claude-sonnet-4-6）， max_tokens 為 600；客戶端自環境讀取 ANTHROPIC_API_KEY 與 ANTHROPIC_BASE_URL。
  • NullExtractor：當 ODM_EXTRACT_KNOWLEDGE=false 時回傳空知識。

輸出契約

系統提示要求模型「只」輸出一個 JSON 物件。_parse 會剝除 Markdown code fence、 擷取首尾大括號之間的 JSON，並正規化三個欄位：

{
  "summary":   "一到兩句，描述這段內容的主旨",
  "concepts":  ["1–8 個它引入的領域術語或實體"],
  "relations": ["以 A -> B 描述關係，可為空"]
}

若輸出不含 JSON 物件，則丟出 ExtractionError。萃取結果會寫入中繼資料， 並由 Chunk.embedding_text() 折入嵌入文字，以增強檢索的語意。

09向量儲存

ChromaStore 封裝 Chroma。集合本身不掛載 embedding function（向量一律由本層提供）， 採用 cosine 空間（hnsw:space）。預設持久化於 <data_dir>/chroma；測試時可注入 EphemeralClient。

方法行為
upsert(chunks)以 chunk.id（內容雜湊）冪等寫入，嵌入的是 embedding_text()；同批會先以 id 去重，避免重複內容觸發 DuplicateIDError。
search(query, top_k, where, mode, source_contains)向量或混合檢索，詳見第 17 節。
get_ids_for_source / delete_ids / get_all_sources供變更感知同步（第 16 節）使用的來源查詢與批次刪除。
get_items / get_item分頁瀏覽，或取得單筆。
update_metadata / delete_item就地更新中繼資料，或刪除（對象不存在時回傳 False）。
list_collections / create_collection / drop_collection知識庫（集合）管理，詳見第 19 節。
stats()回傳 collection、count、embedder、dim。
clear()刪除並重建集合。

14測試策略

測試全程離線：以確定性的 FakeEmbedder（用雜湊式 bag-of-words，讓詞彙重疊決定 cosine 相似度）、 FakeExtractor 與 Chroma EphemeralClient 取代真實相依，既不需網路，也不需下載模型。

  • 單元測試：語法樹切塊應落在函式與類別邊界，符號與行號需正確；文字切塊需符合大小與重疊；loader 的路由與二進位失敗。
  • 整合測試：攝取一個小型 fixture，驗證可被檢索，且重複攝取為冪等（靠雜湊去重）。
  • API 測試：以 httpx 或 TestClient 驗證 stats、search、items 增刪查改，以及 ingest 的 SSE 串流。
pytest   # 全數通過，無網路、無模型下載

15擴充指南

  • 新增程式語言：於 loader.LANGUAGE_BY_EXT 補上副檔名對應， 並在 code_splitter._GRAMMARS 註冊該語言的 grammar 模組與工廠函式（同時把對應 wheel 加入相依）。

16變更感知同步與並行萃取

變更感知同步

片段 id 為內容雜湊，因此「未變動」內容重新攝取本即冪等。但檔案內容變動後， 舊片段（舊行號與內文）會殘留為孤兒；刪除的檔案也會留下過時片段。為此，逐檔處理時會做一次對帳：

  • 算出本次該檔的新 id 集合，向 get_ids_for_source(source) 取得既有 id， 以 delete_ids(existing − new) 移除孤兒，再 upsert。新增、更新、移除一次到位。
  • 目錄同步：ingest_path(dir, sync=True) 記錄本次走訪到的所有來源， 再比對 get_all_sources()，移除位於該根目錄下、但本次未見（即已刪除）的檔案片段。
  • 移除數量計入 IngestReport.chunks_pruned，並發出 prune 事件（明確回報）。

並行萃取

知識萃取是 IO 密集（Anthropic 呼叫）。_extract_all 以 ThreadPoolExecutor 並行處理每片段（執行緒數由 ODM_EXTRACT_CONCURRENCY 控制，預設 8），結果直接寫回各 Chunk 物件，順序無關；單一片段失敗只記入 report.errors，不中斷整體。 嵌入仍於單次 upsert 批次完成。

17混合檢索與過濾

store.search(query, top_k, where, mode, source_contains) 預設 mode="hybrid"： 同時取得密集向量與 BM25 結果，再以 Reciprocal Rank Fusion（RRF）融合。 這能補足嵌入對精確符號／識別字（如函式名）易模糊的弱點。

  • 詞彙索引（retrieval/lexical.py 的 LexicalIndex）：以 rank-bm25 建於集合文件之上，採惰性建置與 dirty 標記，於 upsert／刪除後重建。
  • 融合：rrf_fuse(rankings, k=60)，分數 Σ 1/(k + rank + 1)，兩路皆高者勝出。
  • 過濾：build_where 將 kind / language / symbol 轉為 Chroma 精確 where； source_contains 以 Python 後置過濾來源路徑子字串。

以 ODM_SEARCH_MODE=vector 可退回純密集檢索（此模式 score = 1 − cosine_distance）。

當啟用知識合成後，檢索可再以 RRF 融合合成文章——見 §20 知識合成與文章。

18RAG 問答（ask）

query/rag.py 的 answer_question(query, store, settings, top_k) 先檢索最相關片段， 再交由 Claude 合成附 [n] 行內引用的答案，並回傳 {answer, citations[]}。

  • LLM 呼叫隔離於可注入的 synthesize callable，使邏輯可離線單元測試；預設後端用 Anthropic SDK，模型由 ODM_ANSWER_MODEL 指定。
  • 無金鑰（或 SDK／API 失敗）時明確丟出 AnswerError；Web 端回 503， MCP 回 {error}，CLI 印出錯誤——絕不悄悄回空答案。
  • 無檢索結果時直接短路，不呼叫模型。

19多知識庫（collections）

作用中的集合可於執行期選擇，而非綁定單一 collection_name。 build_context(collection=...) 為每個知識庫建立並快取一份 Context。

  • 管理：list_collections()（含片段數）、create_collection、 drop_collection，皆透過共用的 Chroma client；同一 data_dir 的多個 store 共享底層 DB。
  • 選擇：Web 以 collection 查詢參數或 X-Collection 標頭指定； CLI 以 --collection；MCP 工具以 collection 參數。未指定則用預設。
  • 前端：側欄的知識庫切換器可建立／切換，選擇存於 localStorage 並隨每次請求送出。

20知識合成與文章（synthesis）

synthesis/ 的 synthesize_articles(store, settings, *, graph, limit, dry_run) 把零散片段升華為跨片段、具商業意義的文章（Article），存入並列集合 {collection}__articles。按需以 CLI synthesize 觸發，非攝取階段、無自動排程。

  • 主題探勘（topics.py）：從片段 concepts 蒐集候選，套結構閘門——須 cross-validated（程式碼與文件皆現）或 business_hits > 1 才入選。
  • 撰寫 + 評審雙閘門（llm.py）：ArticleWriter 產出帶 [n] 引用的文章， ArticleCritic 再評 grounded 與 business_meaningful，兩者皆真才保留。
  • 冪等：Article.id = sha256(topic + sorted(source_chunk_ids))，重跑不產生重複； Article duck-type 片段儲存介面（kind="article"）。
  • 文章增強檢索（retrieval/unified.py 的 search_unified）：以 RRF 將文章與片段融合； 旗標 retrieve_include_articles（預設 on）控制，/api/search、/api/ask 皆走此路徑。
  • 介面：CLI synthesize [--limit] [--dry-run]、GET /api/articles、Web Articles 頁。
